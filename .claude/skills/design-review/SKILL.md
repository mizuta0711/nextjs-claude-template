# 設計レビュー

実装に入る前に、設計書間の矛盾をチェックしてください。

## Step 1: レビュー範囲の絞り込み

**全設計書を読まないこと。** 関連する設計書のみをチェックする。

$ARGUMENTS が指定されている場合はそのファイルに関連する設計書のみ。
未指定の場合は、直近の未コミット変更 or 直近コミットから関連設計書を特定:

```bash
git diff --name-only HEAD~1
```

### 変更ファイル → チェック対象の対応表

| 変更内容 | 読むべき設計書 |
|---------|-------------|
| `src/app/api/` | `docs/設計書/API一覧.md` + 対応表 |
| `src/lib/services/` | `docs/設計書/サービス・リポジトリ一覧.md` + 対応表 |
| `src/features/*/hooks/` | `docs/設計書/フック一覧.md` |
| `prisma/schema.prisma` | `docs/設計書/テーブル定義書.md` |
| `src/lib/ai/prompts/` | `docs/設計書/ai-prompt-design.md` |
| UI/ページの新規追加 | 既存ページ1つを読んでレイアウト比較 |

**API設計の詳細仕様** (`.claude/01_development_docs/03_api_design.md`) は API ルート変更時のみ読む。

## Step 2: チェック実施

該当する項目のみチェックする（全項目を毎回チェックしない）。

### API関連の変更時
- リクエスト・レスポンスのJSON例が定義されているか
- `API一覧.md` と `03_api_design.md` の内容が一致しているか
- ページネーションキーが `items` に統一されているか

### DB関連の変更時
- APIレスポンスのフィールドがDBスキーマに存在するか
- 型が一致しているか

### サービス/フック変更時
- 対応表に全エンドポイントが記載されているか
- サービスのメソッドがAPI設計をカバーしているか

### UI/画面の変更時
- 既存レイアウトとの一貫性
- レスポンシブ対応
- コンポーネント再利用・用語統一

## Step 3: 結果報告 + ファイル保存

レビュー結果を `docs/reviews/YYYYMMDD_HHMMSS_design-review.md` に保存:

```markdown
# 設計レビュー結果

- 実施日時: YYYY-MM-DD HH:MM
- 対象: （チェック対象の概要）

## チェック範囲
- （読んだ設計書の一覧）

## 要修正
| # | 重要度 | 内容 | 対象ファイル | 対応状況 |
|---|--------|------|------------|---------|

## 判定: ✅ 承認 / ⚠️ 条件付き承認 / ❌ 差し戻し
```

ユーザーには判定と要修正の件数のみ報告（詳細はファイル参照）。
