# 設計レビュー

実装に入る前に、設計書間の矛盾をチェックしてください。

## チェック対象ドキュメント

以下の設計書を全て読み、整合性を確認:

1. `.claude/01_development_docs/03_api_design.md` — API定義
2. `docs/設計書/API一覧.md` — API定義（docs側）
3. `docs/設計書/サービス・リポジトリ一覧.md` — サービス設計
4. `docs/設計書/フック一覧.md` — フック設計
5. `docs/設計書/API・サービス・リポジトリ・フック対応表.md` — 横断対応
6. `docs/設計書/テーブル定義書.md` — DB設計
7. `docs/設計書/ai-prompt-design.md` — AIプロンプト設計
8. `docs/設計書/チャット・ナレッジフロー図.md` — フロー図
9. `prisma/schema.prisma` — 実際のDBスキーマ

$ARGUMENTS が指定されている場合は、そのファイルに関連する設計書のみをチェック。

## チェック項目

### 1. API定義の完全性
- 全エンドポイントにリクエスト・レスポンスのJSON例が定義されているか
- `03_api_design.md` と `docs/設計書/API一覧.md` の内容が一致しているか
- ページネーションのレスポンスキーが `items` に統一されているか

### 2. API ↔ DB の整合性
- APIレスポンスのフィールドがDBスキーマ（prisma/schema.prisma）に存在するか
- 型が一致しているか

### 3. API ↔ サービス ↔ フック の整合性
- 対応表に全エンドポイントが記載されているか
- サービスのメソッドがAPI設計をカバーしているか
- フック一覧が最新か

### 4. フロー図の最新性
- チャット・ナレッジフロー図が実装と一致しているか

## レビュー結果の保存（必須）

レビュー結果を以下のファイルに保存すること:

**ファイルパス**: `docs/reviews/YYYYMMDD_HHMMSS_design-review.md`（実行時の日時）

**ファイル内容**:
```markdown
# 設計レビュー結果

- 実施日時: YYYY-MM-DD HH:MM
- 対象: （チェック対象の概要）

## 問題なし
- （チェック項目）

## 要修正
| # | 重要度 | 内容 | 対象ファイル | 対応状況 |
|---|--------|------|------------|---------|
| 1 | 高/中/低 | （問題の説明） | （ファイルパス） | 未対応/対応済み |

## 判定: 承認 / 条件付き承認 / 差し戻し
```

レビュー結果はユーザーにも報告すること。
