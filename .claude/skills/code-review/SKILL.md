# コードレビュー＋指摘対応

## Step 1: レビュー対象の特定

$ARGUMENTS が指定されている場合はそのファイル/ディレクトリを対象。
未指定の場合は直近の未コミット変更を対象:

```bash
git diff --name-only
```

変更がない場合は直近1コミットの変更ファイル:

```bash
git diff --name-only HEAD~1
```

## Step 2: レビュー実施

### 必須チェック項目

#### 1. API設計書との突き合わせ（API関連の変更時）
- `.claude/01_development_docs/03_api_design.md` を読み、APIルートのレスポンスが設計書と一致しているか確認
- フロントエンドのAPI呼び出しコード（hooks等）が設計書のレスポンス形式を正しく参照しているか
- **ページネーションキーが `items` に統一されているか**

#### 2. 型安全性
- `any` 型の使用がないか
- API レスポンスの型定義が正しいか（`src/types/index.ts` と一致）
- Zodスキーマとレスポンス型の整合性

#### 3. エラーハンドリング
- API呼び出しのtry/catch
- ユーザーフレンドリーなエラーメッセージ
- AI応答パース失敗時のフォールバック

#### 4. Zustandストアとの整合性
- API取得データがストアに正しく同期されているか
- フック内でストアに書き込んでいるか（直接fetchせずフック経由か）

#### 5. セキュリティ
- withAuthによる認証チェック
- userIdスコープの確認（他ユーザーのデータにアクセスできないか）

#### 6. パフォーマンス
- 不要なAPI呼び出しがないか
- N+1クエリがないか

## Step 3: 結果報告

```

- 内容: （100文字以内）
- 対応: 修正済み / 要修正

#### [中程度] 指摘2
- ...

### 良かった点
- ...

### 判定: ✅ 承認 / ⚠️ 条件付き承認 / ❌ 差し戻し
```

## Step 4: レビュー結果の保存（必須）

レビュー結果を以下のファイルに保存すること:

**ファイルパス**: `docs/reviews/YYYYMMDD_HHMMSS_code-review.md`（実行時の日時）

**ファイル内容**:
```markdown
# コードレビュー結果

- 実施日時: YYYY-MM-DD HH:MM
- 対象: （レビュー対象ファイル一覧）

## 指摘事項
| # | 重要度 | 場所 | 内容 | 対応状況 |
|---|--------|------|------|---------|
| 1 | 重要/中/低 | ファイル:行 | （指摘内容） | 修正済み/未対応 |

## 良かった点
- （良い実装）

## 判定: ✅ 承認 / ⚠️ 条件付き承認 / ❌ 差し戻し
```

## Step 5: 指摘対応（自動修正）

重要度が「重要」の指摘で、明確な修正方法がある場合は自動的に修正を行う。
修正後、レビュー結果ファイルの対応状況を「修正済み」に更新する。

修正が複雑な場合や判断が必要な場合は、修正案を提示してユーザーに確認を求める。
