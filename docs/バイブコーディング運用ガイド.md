# Claude Code によるバイブコーディング運用ガイド

本テンプレートで実践しているClaude Code活用のノウハウをまとめる。
実プロジェクト（Next.js + Prisma + TailwindCSS）での運用を通じて得た知見に基づく。

---

## 1. テンプレートの全体構成

```
.claude/
├── agents/              # カスタムエージェント定義
│   ├── coding-specialist.md    # 実装担当
│   ├── code-reviewer.md        # レビュー担当
│   ├── documentation-manager.md # ドキュメント管理
│   └── product-advisor.md      # 企画・UXアドバイザー
├── skills/              # スラッシュコマンド（カスタムスキル）
│   ├── build-check/     # /build-check — ビルド+lint一括実行
│   ├── code-review/     # /code-review — コードレビュー
│   ├── design-review/   # /design-review — 設計書間の矛盾チェック
│   ├── update-docs/     # /update-docs — 設計書の自動更新
│   ├── sync-check/      # /sync-check — 設計書と実装の全量照合
│   ├── complete-feature/ # /complete-feature — 機能完了処理
│   ├── pre-push-check/  # /pre-push-check — push前の同期チェック
│   └── new-feature/     # /new-feature — 設計書テンプレートから新規作成
├── 01_development_docs/ # 設計方針・パターン・ルール（低頻度変更）
├── 02_design_system/    # デザインシステム方針
└── 03_library_docs/     # ライブラリ利用ガイド

docs/
├── 設計書/              # 実態の一覧・定義（高頻度変更、コードと同期）
├── features/            # 機能設計書
└── reviews/             # レビュー結果

CLAUDE.md                # 全エージェント共通のルール・フロー定義
```

## 2. 工夫のポイント

### 2-1. 変更規模別の開発フロー（S / M / L）

最大の工夫。当初は全変更に9ステップの必須フローを課していたが、**小さな修正でも重いプロセスを要求するため形骸化**した。変更規模に応じて3段階のフローを定義することで解決した。

| 規模 | 対象 | フロー |
|------|------|--------|
| **S** | バグ修正、テキスト修正、1ファイル程度 | 実装 → `/build-check` → コミット |
| **M** | 機能追加、API変更、複数ファイル | 設計 → 実装 → `/code-review` → `/build-check` → `/update-docs` → コミット |
| **L** | 新機能、大規模変更、DBスキーマ変更 | 設計 → `/design-review` → 実装 → `/code-review` → `/build-check` → `/update-docs` → コミット |

**迷ったらMを使う**のがルール。Sで済む作業をMで実行しても害はないが、Lが必要な作業をSで済ませると設計書との乖離が発生する。

### 2-2. 設計書駆動開発とドキュメント同期の自動化

Claude Codeに設計書を先に読ませることで、実装の品質が劇的に向上する。ただし**設計書が古いと逆効果**になるため、同期の仕組みが不可欠。

#### 同期を支える3つのスキル

| スキル | 役割 | 実行タイミング |
|--------|------|---------------|
| `/update-docs` | 変更駆動の差分同期。コミット前に実行し、変更されたファイルに対応する設計書を更新 | 毎コミット前 |
| `/sync-check` | 全量照合。実装と設計書を網羅的に突き合わせ、`/update-docs`では検出できない乖離を発見 | 機能完了時 |
| `/pre-push-check` | push前にコミットハッシュが `.doc-sync.md` に記録済みか確認 | 毎push前 |

`/update-docs` で全タスク完了を検知すると、自動で `/sync-check` → `completed/` 移動が実行される。これにより「設計書を更新し忘れたままpush」を防止する。

#### ドキュメントの二重管理防止

実運用で最も発生しやすい問題が**同じ情報を2箇所に書いて片方だけ更新する**パターン。これを防ぐため、ドキュメントの役割を明確に分離している。

| 場所 | 役割 | 例 |
|------|------|-----|
| `.claude/` 配下 | **方針・パターン・ルール** | 「API命名はケバブケース」「ページネーションはcursor方式」 |
| `docs/設計書/` | **実態の一覧・定義** | 「GET /api/v1/users のレスポンスは {...}」 |

ルールと実態を分けることで、同期が必要なのは `docs/設計書/` だけになる。

### 2-3. エージェント定義の最適化

#### 共通ルールの一元化

当初、各エージェント定義に同じコーディングルール（`any`禁止、`memo<Props>`パターン等）を重複して書いていた。これは以下の問題を引き起こした:

- **トークンコスト増大**: エージェント起動時に全文がコンテキストに読み込まれる
- **ルール変更時の更新漏れ**: 1箇所だけ直して他を忘れる

解決策として、共通ルールを `CLAUDE.md` の「コーディングルール（全エージェント共通）」に一元化し、各エージェント定義からは「CLAUDE.md のルールに従うこと」と参照する形にした。

#### エージェント定義は短く

各エージェント定義は **50行前後** に圧縮している。含めるのは:

- 役割の1行説明
- 参照ドキュメントの対応表（作業内容→参照先）
- そのエージェント固有のプロセス
- 完了報告の必須事項

以下は**含めない**:
- TypeScriptコード例（良い例/悪い例）
- mermaid図
- チェックリストの詳細

これらはCLAUDE.mdの共通ルールや `.claude/01_development_docs/` の設計方針ドキュメントに書けば十分。

#### PMエージェントは不要

当初、project-manager エージェントを「司令塔」として設置し、全作業をサブエージェントに委任する設計にしていた。しかし:

- PM→サブエージェントの中間委任でトークンコストが2倍になる
- コンテキストの受け渡しが不正確になる
- ユーザーが直接 coding-specialist を呼ぶ方が速く正確

結局PMは削除した。タスク分解が必要な場合はClaude Code組み込みの Plan 機能で十分。

### 2-4. サブエージェント運用の実践知

#### コスト最適化

- サブエージェントは原則 `model: "sonnet"` で起動する（CLAUDE.md + agents の frontmatter で設定）
- Opus を使うのは複雑な設計判断やアーキテクチャ決定の場合のみ
- 簡単な作業（grep、設計書更新、定型編集）は積極的に sonnet に委譲

#### サブエージェントの結果は信用しすぎない

実運用で学んだ最も重要な教訓:

- **ビルド成功 ≠ 正しい実装**。ロジックの正しさはメインで検証する
- サブエージェントの結果は **必ずメインで差分確認・フロー検証してからコミット** する
- `router.push` / `fetch` など副作用のあるコードは遷移先・呼び出し先の挙動まで追跡検証する

#### BE/FE分離時のAPI型契約

バックエンドとフロントエンドを別のサブエージェントに委譲すると、型の不一致が発生しやすい。対策として:

1. 実装前に `src/types/` に共有レスポンス型を定義する
2. route.ts と hooks の両方がその型を import する
3. レビュー時にAPI提供側と消費側の型を突き合わせる

### 2-5. 機能設計書テンプレートによるタスク管理

`/new-feature 機能名` で設計書テンプレートが自動生成される。設計書は以下の構成:

```
メタ情報（ステータス: 🔵設計中 → 🟡実装中 → 🟢完了）
├── タスク一覧（ステータス管理: 🔵/🟡/✅/⏸️/❌）
├── 設計詳細（フェーズごと）
│   ├── 概要
│   ├── 作成・変更ファイル一覧
│   ├── 設計詳細
│   └── 受け入れ条件
└── 改訂履歴
```

この設計書が「進捗ダッシュボード」として機能し、Claude Codeが `/update-docs` 実行時にタスクステータスを自動更新する。全タスク完了を検知すると sync-check → completed/ 移動まで自動実行される。

### 2-6. テーブル定義書の自動生成

`prisma/schema.prisma` の `///` コメントから `docs/設計書/テーブル定義書.md` を自動生成するスクリプト（`tools/scripts/generate-table-docs.ts`）を用意している。

手動でテーブル定義書を書くと乖離が最も発生しやすいため、自動生成に切り替えた。スキーマ変更時は自動生成スクリプトを再実行するだけで同期が取れる。

### 2-7. 設計書優先の原則

既存コードの把握は `docs/設計書/` を先に読む。ソースコードは設計書で不足がある場合のみ。

これにより:
- トークン消費を抑えられる（設計書はソースより短い）
- 全体像を素早く把握できる
- Explore エージェントに「徹底的に調査」のような指示を出す必要がなくなる

## 3. 実運用で踏んだ落とし穴と対策

### 落とし穴1: 「省略禁止」は守られない

**問題**: 全変更に9ステップ必須のフローを定義し「省略禁止」と明記していたが、1行のバグ修正でも全ステップを実行するのは現実的でなく、次第に無視された。

**対策**: S/M/L の3段階フローに変更。Sは「実装→ビルド→コミット」の3ステップのみ。

### 落とし穴2: 設計書と実装の乖離

**問題**: `/update-docs` を毎回実行していても、「そもそも設計書に記載がない」実装は検出できない。変更駆動の同期だけでは不十分だった。

**対策**: 全量照合の `/sync-check` を導入し、`/update-docs` の全タスク完了時に自動実行する仕組みにした。

### 落とし穴3: エージェント定義の肥大化

**問題**: 各エージェント定義に詳細なTypeScriptコード例、mermaid図、チェックリストを書いた結果、1ファイル500行超になり、エージェント起動のたびに大量のトークンを消費していた。

**対策**: 共通ルールをCLAUDE.mdに一元化し、各エージェント定義を50行前後に圧縮。合計約1,600行 → 約160行（90%削減）。

### 落とし穴4: サブエージェントの品質を過信

**問題**: サブエージェントが「実装完了」と報告した内容をそのままコミットしたところ、ビルドは通るが実際の動作が設計と異なるケースがあった。

**対策**: 「ビルド成功 ≠ 正しい実装」をルール化。サブエージェントの結果はメインで必ず差分確認・フロー検証する。BE/FEを別サブエージェントに委譲する場合は事前に共有型定義を作成する。

### 落とし穴5: ドキュメントの二重管理

**問題**: API設計ルール（`.claude/01_development_docs/03_api_design.md`）と個別API定義（`docs/設計書/API一覧.md`）の両方にエンドポイント情報を書いてしまい、片方だけ更新されて矛盾が発生。

**対策**: ドキュメントの役割を「方針（How）」と「実態（What）」に明確分離。同じ情報を2箇所に書かないルールを CLAUDE.md に明記。

## 4. テンプレートの導入方法

1. このリポジトリをクローンまたはコピー
2. `CLAUDE.md` のプロジェクト概要・スタック情報を自分のプロジェクトに合わせて編集
3. `.claude/01_development_docs/` 内の設計方針ドキュメントをプロジェクトに合わせてカスタマイズ
4. 不要な設計書（AI関連、認証関連等）があれば削除
5. `/new-feature 機能名` で最初の機能設計書を作成して開発開始

## 5. まとめ

| 観点 | このテンプレートのアプローチ |
|------|---------------------------|
| プロセス設計 | 変更規模に応じた S/M/L フローで、過剰なプロセスを回避 |
| 品質担保 | スキルによるレビュー・ビルド・設計書同期の自動化 |
| コスト最適化 | サブエージェントは sonnet、エージェント定義は最小限 |
| ドキュメント管理 | 方針と実態を分離し、同期はスキルで自動化 |
| 信頼性 | サブエージェントの結果は必ず検証、型契約で BE/FE の整合性を担保 |

バイブコーディングは「AIに全部任せる」ことではなく、**AIが最大限パフォーマンスを発揮できる仕組みを整えたうえで、人間が判断・検証する**ことで成立する。このテンプレートはその仕組みの実装例である。
