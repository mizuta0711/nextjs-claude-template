# Claude Code サンドボックス環境構築ガイド

Windows 上で Claude Code を安全かつ快適に利用するための、WSL2 + Dev Container (Docker) 構成の解説ドキュメントです。Next.js アプリ開発と Playwright によるブラウザ自動テストを想定しています。

## この構成でできること

- **Claude Code CLI の利用**：コンテナ内で `claude` コマンドを安全に実行できる
- **VS Code Claude Code 拡張での開発**：VS Code の Dev Containers 拡張経由で、コンテナ内の Claude Code 拡張を普段通りに使える
- **`--dangerously-skip-permissions` の安全な使用**：万が一破壊的な操作が走っても影響はコンテナ内に閉じる
- **Next.js アプリのホットリロード開発**：Windows 側ブラウザから `http://localhost:3000` で動作確認可能
- **Playwright による E2E テスト / ブラウザ自動化**：ヘッドレスブラウザでのテスト実行、MCP 経由での Claude による自律的ブラウザ操作
- **プロジェクトごとの環境分離**：プロジェクト別に Node.js バージョンや依存パッケージを分離できる
- **環境のコード化**：`.devcontainer/devcontainer.json` で環境を Git 管理でき、再現性が高い

## なぜこの構成なのか

### 他方式との比較

| 方式 | 隔離性 | 起動速度 | リソース消費 | 開発体験 |
|------|--------|----------|--------------|----------|
| WSL2 単体 | △ | ◎ | ◎ | ◎ |
| **WSL2 + Dev Container** | **◯** | **◯** | **◯** | **◎** |
| VirtualBox 等の仮想PC | ◎ | × | × | △ |

**WSL2 単体**は `/mnt/c/` 経由で Windows のファイルに直接アクセスできてしまうため、サンドボックスとしての隔離性が弱い。

**仮想PC方式**は最も隔離性が高いが、起動・リソース・ホスト連携すべてで開発体験が落ちる。

**WSL2 + Dev Container** は、WSL2 が Linux 環境を、Docker が隔離を提供する二層構造。開発体験を損なわずに実用的な隔離を実現できる。

## 構成全体像

```
┌─────────────────────────────────────────────┐
│ Windows (ホストOS)                          │
│                                             │
│  ┌───────────────┐   ┌──────────────────┐   │
│  │  VS Code      │   │ ブラウザ(Chrome) │   │
│  │  (UI部分)     │   │ localhost:3000   │   │
│  └───────┬───────┘   └────────┬─────────┘   │
│          │                    │             │
│  ┌───────▼────────────────────▼─────────┐   │
│  │  WSL2 (Ubuntu)                       │   │
│  │  ┌────────────────────────────────┐  │   │
│  │  │  Docker Dev Container          │  │   │
│  │  │  - VS Code Server              │  │   │
│  │  │  - Claude Code CLI / 拡張      │  │   │
│  │  │  - Node.js, Next.js, Playwright│  │   │
│  │  │  - プロジェクトコード          │  │   │
│  │  └────────────────────────────────┘  │   │
│  └──────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

**ポイント**：VS Code の画面は Windows 側で表示されるが、エディタ操作・ターミナル・Claude Code 拡張機能はすべてコンテナ内で動作する。

## 前提環境・必要なもの

### Windows 側

- Windows 10 (バージョン 2004 以降) または Windows 11
- WSL2 (Ubuntu 推奨)
- Docker Desktop for Windows (WSL2 統合を有効化)
- Visual Studio Code
- VS Code 拡張機能：
  - **Dev Containers** (Microsoft 製)

### Docker Desktop 側

- WSL2 ベースのエンジンを有効化
- WSL 統合で対象の Ubuntu ディストリビューションを有効化

### 推奨リソース設定

WSL2 の `.wslconfig` (Windows ユーザーフォルダ直下、例: `C:\Users\Katsuya\.wslconfig`)：

```ini
[wsl2]
memory=8GB
processors=4
```

ブラウザ自動化を行う場合はメモリ 4GB 以上を推奨。

## セットアップ手順

### 1. WSL2 + Docker Desktop のインストール

```powershell
# PowerShell (管理者権限)
wsl --install
```

Docker Desktop を[公式サイト](https://www.docker.com/products/docker-desktop/)からダウンロードしてインストール。起動後、設定で以下を確認：

- `Settings` → `General` → `Use the WSL 2 based engine` が有効
- `Settings` → `Resources` → `WSL Integration` で対象ディストリビューションが ON

### 2. VS Code と拡張機能のインストール

- VS Code 本体をインストール
- 拡張機能「**Dev Containers**」(識別子: `ms-vscode-remote.remote-containers`) をインストール

### 3. プロジェクトの配置場所

**重要**：プロジェクトは WSL2 のホームディレクトリ配下に置く。

```bash
# WSL2 内で
mkdir -p ~/projects
cd ~/projects
```

`/mnt/c/` 配下に置くと：
- ファイル I/O が大幅に遅くなる
- Dev Container から Windows 側へアクセス可能になってしまいサンドボックスの意味が薄れる

### 4. プロジェクトの初期化

```bash
cd ~/projects
npx create-next-app@latest my-app
cd my-app
mkdir .devcontainer
```

### 5. `.devcontainer/devcontainer.json` の作成

次のセクションを参照。

### 6. コンテナで開く

VS Code でプロジェクトフォルダを開き、右下に表示される「Reopen in Container」をクリック。初回はイメージのビルドに数分かかる。

## devcontainer.json の設定例

### 基本構成（Next.js + Claude Code + Playwright）

`.devcontainer/devcontainer.json`:

```json
{
  "name": "nextjs-claude-sandbox",
  "image": "mcr.microsoft.com/devcontainers/javascript-node:20",

  "features": {
    "ghcr.io/devcontainers/features/git:1": {}
  },

  "forwardPorts": [3000],
  "portsAttributes": {
    "3000": {
      "label": "Next.js Dev Server",
      "onAutoForward": "notify"
    }
  },

  "customizations": {
    "vscode": {
      "extensions": [
        "anthropic.claude-code",
        "dbaeumer.vscode-eslint",
        "esbenp.prettier-vscode",
        "ms-playwright.playwright"
      ],
      "settings": {
        "terminal.integrated.defaultProfile.linux": "bash"
      }
    }
  },

  "postCreateCommand": "npm install -g @anthropic-ai/claude-code && npx -y playwright install --with-deps chromium",

  "remoteUser": "node",

  "workspaceFolder": "/workspace",
  "workspaceMount": "source=${localWorkspaceFolder},target=/workspace,type=bind",

  "runArgs": ["--ipc=host"]
}
```

### 設定項目の解説

| 項目 | 目的 |
|------|------|
| `image` | ベースイメージ。Node.js 20 入りの公式イメージを使用 |
| `forwardPorts` | Windows 側からアクセス可能にするポート |
| `extensions` | コンテナ起動時に自動インストールされる VS Code 拡張 |
| `postCreateCommand` | 初回作成時に実行するセットアップコマンド |
| `remoteUser` | コンテナ内で使用するユーザー（非 root 推奨） |
| `workspaceMount` | ワークスペースのマウント設定。**これ以外はマウントしない** |
| `runArgs: ["--ipc=host"]` | Chromium の安定動作に必要 |

## Next.js 開発時の設定

### package.json の修正

コンテナ外からアクセスできるよう、`0.0.0.0` で待ち受けるように設定：

```json
{
  "scripts": {
    "dev": "next dev -H 0.0.0.0",
    "build": "next build",
    "start": "next start -H 0.0.0.0"
  }
}
```

### 開発フロー

1. VS Code のターミナルを開く（コンテナ内のシェルになっている）
2. `npm run dev` を実行
3. VS Code が自動でポートをフォワード、通知が表示される
4. Windows 側のブラウザで `http://localhost:3000` にアクセス
5. ファイル編集 → 保存 → ホットリロード、という普段通りの開発体験

### 複数ポートが必要な場合

```json
"forwardPorts": [3000, 3001, 5432]
```

### 起動時にブラウザを自動で開く

```json
"portsAttributes": {
  "3000": {
    "label": "Next.js",
    "onAutoForward": "openBrowser"
  }
}
```

## Playwright によるブラウザ自動テスト

### 方法1: 従来型の E2E テスト自動実行

すでに Playwright を利用中であれば、そのまま Dev Container 内で実行できる。

```bash
# コンテナ内で
npm run test:e2e
```

Claude Code に以下のような指示が可能：

> コード変更後、`npm run test:e2e` を実行してください。失敗したら結果を読んで修正し、再度実行してください。

`--dangerously-skip-permissions` と組み合わせれば、Claude Code が「コード修正 → テスト実行 → 結果確認 → 再修正」のループを自律的に回す。

### 方法2: Playwright MCP を使う（Claude 自身がブラウザ操作）

Microsoft 公式の **Playwright MCP** を Claude Code に登録すると、Claude が自分でブラウザを立ち上げて動作確認できる。

```bash
# コンテナ内で Claude Code に MCP を登録
claude mcp add playwright npx '@playwright/mcp@latest'
```

できること：
- ページ遷移、クリック、フォーム入力などの操作
- スクリーンショット取得と Claude による視覚的確認
- コンソールエラーやネットワークエラーの読み取り
- 「UI 実装 → 動作確認 → 問題修正」の自律ループ

### セットアップの注意点

**Playwright ブラウザのインストール**は `postCreateCommand` に含めておくと便利：

```json
"postCreateCommand": "npm install -g @anthropic-ai/claude-code && npx -y playwright install --with-deps chromium"
```

**Chromium の安定動作**のため `runArgs` に `--ipc=host` を必ず追加：

```json
"runArgs": ["--ipc=host"]
```

Chromium は `/dev/shm` を多用するが、デフォルトのコンテナでは 64MB しかなくクラッシュの原因になるため。

**メモリ確保**：ブラウザ自動化は比較的メモリを消費するので、WSL2 に 4GB 以上を割り当てる（`.wslconfig` 参照）。

## セキュリティ上の注意点

### `--dangerously-skip-permissions` の影響範囲

デフォルトの Dev Container 構成では、Claude Code の影響範囲は以下に限定される：

- ✅ 影響あり：マウントされたワークスペースディレクトリ (`/workspace`)
- ❌ 影響なし：WSL2 のホームディレクトリ全体
- ❌ 影響なし：`/mnt/c/` 以下の Windows 側ファイル
- ❌ 影響なし：他の WSL ディストリビューション

### やってはいけない設定

以下を行うとサンドボックスが実質的に機能しなくなる：

#### 1. `/mnt/c` のマウント（絶対NG）

```json
// 危険な例
"mounts": [
  "source=/mnt/c/Users/Katsuya,target=/host,type=bind"
]
```

#### 2. Docker socket のマウント

```json
// 危険な例
"mounts": [
  "source=/var/run/docker.sock,target=/var/run/docker.sock,type=bind"
]
```

コンテナ内から特権付きで別コンテナを起動可能になり、実質的にホストへ脱出できる。

#### 3. `--privileged` や `--network=host`

```json
// 危険な例
"runArgs": ["--privileged", "--network=host"]
```

隔離が大きく弱まる。

### 推奨設定チェックリスト

- [ ] `workspaceMount` 以外の `mounts` は追加しない
- [ ] `/mnt/c`、`/`、`~` (ホーム全体) をマウントしない
- [ ] Docker socket を渡さない
- [ ] `--privileged` を付けない
- [ ] `remoteUser` を非 root (`node`、`vscode` など) に設定
- [ ] プロジェクトは WSL2 ホーム配下 (`~/projects/`) に配置
- [ ] 機密情報（APIキーなど）は `.env` で管理し、`.gitignore` に追加

## よくあるハマりどころ

### Windows ブラウザから Next.js にアクセスできない

**原因**：Next.js が `localhost` (127.0.0.1) のみで待ち受けている。

**対応**：`package.json` の dev スクリプトに `-H 0.0.0.0` を追加。

```json
"dev": "next dev -H 0.0.0.0"
```

### Playwright が起動しない / Chromium がクラッシュする

**原因**：コンテナの `/dev/shm` 容量不足、または依存ライブラリ不足。

**対応**：
- `runArgs: ["--ipc=host"]` を追加
- `playwright install --with-deps chromium` で依存関係もインストール

### ファイル操作が遅い

**原因**：プロジェクトを `/mnt/c/` 配下に置いている。

**対応**：WSL2 のホーム (`~/projects/` など) に配置し直す。

### Claude Code 拡張がコンテナ内に入らない

**原因**：`devcontainer.json` の `extensions` に追加されていない。

**対応**：`customizations.vscode.extensions` に `"anthropic.claude-code"` を追加して Rebuild Container。

### コンテナの再構築方法

設定変更後は再構築が必要：

- コマンドパレット → `Dev Containers: Rebuild Container`

## 参考：VM方式との比較

| 項目 | WSL2 + Dev Container | VirtualBox 等 |
|------|---------------------|---------------|
| 起動時間 | 数秒 | 数十秒〜数分 |
| メモリ消費 | 動的（必要分のみ） | 固定割当 |
| ディスク消費 | 共有イメージで節約可 | VM ごとに丸ごと |
| ホスト連携 | シームレス | 手間がかかる |
| 隔離性 | 実用十分 | 最も高い |
| 開発体験 | ほぼ通常通り | テンポが落ちる |

**結論**：業務で特に機密性が高いデータを扱う場合を除き、WSL2 + Dev Container がベストバランス。

## まとめ

- **WSL2 + Docker Dev Container** で開発体験を損なわずサンドボックスが実現できる
- **VS Code の Claude Code 拡張**もコンテナ内で通常通り使える
- **Next.js + ブラウザ動作確認**はポートフォワーディングで Windows 側から可能
- **Playwright** は Dev Container 内で問題なく動作し、MCP 経由で Claude 自律テストも可能
- `/mnt/c` や Docker socket のマウントだけは絶対に避ける

安全性と開発効率を両立した Claude Code 開発環境として、この構成を強く推奨。
